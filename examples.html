To understand how a SALVE execution runs, here we present several examples and describe how an execution with them runs.

Example 1: zsh Manifest

Consider the following manifest that handles zsh configuration.

file {
    source  files/zsh/zshrc
    target  $HOME/.zshrc
}

file {
    source  files/zsh/zshrc.zni
    target  $HOME/.zshrc.zni
}

file {
    source  files/zsh/zsh_git
    target  $HOME/.zsh_git
}

file {
    source  files/zsh/zsh_dircolors
    target  $HOME/.zsh_dircolors
}

file {
    source  files/zsh/myfuncs.sh
    target  $HOME/.myfuncs.sh
}

kept in a SALVE repo, under the name of zsh.manifest, with the following directory structure

zsh.manifest
LICENSE
README.md
default_settings.ini
salve.py
src
|-- ...
tests
|-- ...
files
|-- ...
`-- zsh
    |-- myfuncs.sh
    |-- zsh_dircolors
    |-- zsh_git
    |-- zshrc
    `-- zshrc.zni

and the following default_settings.ini, without overrides

[common]
backup_dir=$HOME/.salve/backups
backup_log=$HOME/.salve/backup.log

[file]
action=copy
mode=600
user=$USER
group=$SALVE_USER_PRIMARY_GROUP

[directory]
action=copy
mode=755
user=$USER
group=$SALVE_USER_PRIMARY_GROUP

[manifest]

zsh Manifest: First Block

When run via ./salve -m zsh.manifest, this manifest first specifies

file {
    source  files/zsh/zshrc
    target  $HOME/.zshrc
}

which expands with the defaults to

file {
    source      files/zsh/zshrc
    target      $HOME/.zshrc
    action      copy
    mode        600
    user        $USER
    group       $SALVE_USER_PRIMARY_GROUP
    backup_dir  $HOME/.salve/backups
    backup_log  $HOME/.salve/backup.log
}

This specifies that the file in the salve repo, at $SALVE_ROOT/files/zsh/zshrc, should be copied to $HOME/.zshrc. Its mode should be set to 600, and if running as root, the user and group should be set to the invoking user and that user's primary group (inspecting $SUDO_USER to deduce who that user is). If there was an existing file at $HOME/.zshrc, it should be backed up into $HOME/.salve/backups/$HOME/.zshrc/, with a unique filename generated by a hash of its contents. In the event of a backup, the date, the filename, and the hash are all logged to the backup logfile.

Example 2: vim Manifest

The following manifest handles vim configuration, and includes a directory action.

directory {
    target  $home/.vim/colors
    action  create
}

file {
    source  files/vim/vimrc
    target  $HOME/.vimrc
}

file {
    source  files/vim/solarized.vim
    target  $HOME/.vim/colors/solarized.vim
}

it is used in SALVE repo, under the name of vim.manifest, with the following directory structure

vim.manifest
LICENSE
README.md
default_settings.ini
salve.py
src
|-- ...
tests
|-- ...
files
|-- ...
`-- vim
    |-- solarized.vim
    `-- vimrc

and the following default_settings.ini, without overrides

[common]
backup_dir=/etc/salve/backups
backup_log=/etc/salve/backup.log

[file]
action=copy
mode=644
user=$USER
group=$SALVE_USER_PRIMARY_GROUP

[directory]
action=copy
mode=755
user=$USER
group=$SALVE_USER_PRIMARY_GROUP

[manifest]

Note that SALVE can only create the backup files if ```/etc/salve``` already exists. Therefore, this manifest only works on platforms where that folder exists. It could be created with SALVE running as root, but some users may consider this dangerous without first reading the SALVE source.

vim Manifest: First Block

When run via ./salve -m vim.manifest, this manifest first specifies

directory {
    target  $HOME/.vim/colors
    action  create
}

which expands with the defaults to

directory {
    target      $HOME/.vim/colors
    action      create
    mode        755
    user        $USER
    group       $SALVE_USER_PRIMARY_GROUP
    backup_dir  /etc/salve/backups
    backup_log  /etc/salve/backup.log
}

This specifies that a directory at $HOME/.vim/colors, should be created if it does not already exist, as with mkdir -p. Its mode should be set to 755, and if running as root, the user and group should be set to the invoking user and that user's primary group (inspecting $SUDO_USER to deduce who that user is). 755 is recommended for directories when possible, as it is the least likely value to create error conditions later in the SALVE run. With this action, there is no backup because the operations performed are necessarily nondestructive.

vim Manifest: All Blocks

As a more complete view, a fully expanded manifest is

directory {
    target      $home/.vim/colors
    action      create
    mode        755
    user        $USER
    group       $SALVE_USER_PRIMARY_GROUP
    backup_dir  /etc/salve/backups
    backup_log  /etc/salve/backup.log
}

file {
    source      files/vim/vimrc
    target      $HOME/.vimrc
    mode        644
    user        $USER
    group       $SALVE_USER_PRIMARY_GROUP
    backup_dir  /etc/salve/backups
    backup_log  /etc/salve/backup.log
}

file {
    source      files/vim/solarized.vim
    target      $HOME/.vim/colors/solarized.vim
    mode        644
    user        $USER
    group       $SALVE_USER_PRIMARY_GROUP
    backup_dir  /etc/salve/backups
    backup_log  /etc/salve/backup.log
}

Note that the file[mode] and directory[mode] are distinct, and that the specified directory action overrides the default copy.
